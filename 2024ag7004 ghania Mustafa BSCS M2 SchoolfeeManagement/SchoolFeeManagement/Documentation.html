1. Software Requirements Specification (SRS) / Design Document

1.1 Problem Statement

Many schools still manage fee collection manually — using paper ledgers, handwritten receipts, and spreadsheets. This approach is error-prone, time-consuming, and makes reporting difficult. There is a clear need for a simple, reliable desktop application that allows school administrators to:

  • Maintain an accurate student database
  • Record and track fee payments per student, per term
  • Generate receipts and financial reports instantly
  • Ensure only authorised personnel can access the system

1.2 Objectives

  1. Build a fully functional Windows Forms desktop application in C#
  2. Implement complete CRUD operations on the two main entities: Students and Payments
  3. Provide a secure login system with session management
  4. Offer search/filter capabilities across all data grids
  5. Display a dashboard with real-time statistics
  6. Generate payment receipts and exportable reports
  7. Apply a consistent, professional blue and grey colour theme

3. Form Designs & Descriptions

📸 In your submitted documentation, replace each description below with an actual screenshot of the running form.

Form 1 — Login Screen (LoginForm.cs)

Layout: Split-panel. Left panel (blue): branding, app name, tagline. Right panel (white): username field, password field, Login button, error label, hint text.

Key behaviour: Validates both fields are filled, queries the Users table with parameterised SQL, sets session variables on success, and shows an error label on failure.

Form 2 — Main Dashboard (MainDashboardForm.cs)

Layout: Dark-blue top bar (app title + logged-in user), dark-blue sidebar (navigation buttons), white content area.

Content area on load: Four stat cards (Total Students, Total Payments, Total Revenue, Today's Payments) and a DataGridView showing the 15 most recent payments.

Sidebar buttons: Dashboard, Students, Fee Payment, View Payments, Reports, Logout.

Form 3 — Student Management (StudentManagementControl.cs)

Layout: Header, toolbar (Search box, Add/Edit/Delete/Refresh buttons), DataGridView listing all active students.

Add/Edit mode: The grid hides and an inline form appears with all student fields. Clicking Save inserts/updates the database; Cancel returns to the grid.

Delete: Asks for confirmation before removing the student and their payment records.

Form 4 — Fee Payment (FeePaymentControl.cs)

Layout: Left card (payment form), right card (recent payments grid + receipt preview).

Fields: Student (combo), Fee Type (combo, auto-fills amount), Amount, Payment Method, Term, Academic Year, Receipt Number (read-only auto), Notes.

On Process Payment: Validates all required fields, parses amount with InvariantCulture to avoid comma/period locale bugs, inserts into Payments table, shows receipt preview, refreshes history grid.

Form 5 — View Payments (ViewPaymentsControl.cs)

Layout: Header, toolbar (Search, Term filter, Year filter, Export CSV, Delete buttons), DataGridView.

Filtering: Real-time text search + Term and Year combo filters applied dynamically to the SQL query.

Export: Saves visible rows to a user-specified CSV file.

Form 6 — Reports & Analytics (ReportsControl.cs)

Layout: Header, row of 5 report-type buttons, report title label, DataGridView showing results.

Report types: Revenue by Fee Type, Revenue by Grade, Monthly Revenue, Top Paying Students, Term Summary.

4.2 Design Principles Applied

  • Separation of Concerns: All SQL is in DatabaseHelper; all theme constants are in ThemeHelper; forms only handle UI logic.
  • DRY (Don't Repeat Yourself): ThemeHelper factory methods (CreateStyledGrid, CreateHeader, CreatePrimaryButton) eliminate repeated styling code.
  • Parameterised Queries: Every SQL statement uses SQLiteParameter to prevent SQL injection.
  • Defensive Coding: All critical paths (payment save, login, delete) are wrapped in try/catch with user-facing error dialogs.

6. User Manual

6.1 Logging In

  1. Launch the application — the Login screen appears.
  2. Enter Username: admin and Password: admin123.
  3. Click LOGIN or press Enter.
  4. On success, the Main Dashboard opens.
  5. On failure, a red error message appears beneath the fields.

6.2 Dashboard

The home screen shows four summary cards at the top:

  • Total Students — active students in the database
  • Total Payments — all payment records
  • Total Revenue — sum of all AmountPaid
  • Today's Payments — payments recorded today

Below the cards is a grid showing the 15 most recent payments.

6.3 Managing Students

Click 👥 Students in the sidebar.

  • Add: Click ➕ Add Student → fill in required fields (marked *) → click 💾 Save Student.
  • Search: Type in the search box — the grid filters in real time by name, student number, or grade.
  • Edit: Select a row → click ✏ Edit → modify fields → click 💾 Save Student.
  • Delete: Select a row → click 🗑 Delete → confirm in the dialog. This also removes the student's payment records.

6.4 Recording a Fee Payment

Click 💰 Fee Payment in the sidebar.

  1. Select the Student from the dropdown — grade and parent info appears below.
  2. Select the Fee Type — the standard amount auto-fills in the Amount box.
  3. Adjust the Amount if needed (partial payment is allowed).
  4. Choose Payment Method, Term, and Academic Year.
  5. Optionally add a Note.
  6. Click 💳 Process Payment.
  7. A receipt preview appears on the right, and the recent payments grid refreshes.
  8. Click 🔄 Clear to reset the form for the next payment.

6.5 Viewing & Searching Payments

Click 📋 View Payments in the sidebar.

  • Use the Search box to find by student name, number, or receipt number.
  • Use the Term and Year dropdowns to filter by period.
  • Click 📋 Export to CSV to save filtered results as a spreadsheet.
  • Select a row and click 🗑 Delete to remove a payment record.

6.6 Reports

Click 📊 Reports in the sidebar. Use the report buttons to switch views:

  • Revenue by Fee Type — totals per category
  • Revenue by Grade — breakdown per grade
  • Monthly Revenue — month-by-month trend
  • Top Payers — ranked student list
  • Term Summary — per-term totals

6.7 Logging Out

Click 🚪 Logout at the bottom of the sidebar → confirm in the dialog → the Login screen returns. Session data is cleared immediately.

7. Challenges & Solutions

Challenge 1 — Amount Parsing Bug (Fee Payment)

Problem: When a fee type was selected, the standard amount was auto-filled using .ToString("N2") (e.g. "5,000.00"). double.TryParse with the default culture failed on the comma separator, causing every payment submission to show "Invalid amount."

Solution: Changed the amount text box population to use .ToString("F2", CultureInfo.InvariantCulture) (e.g. "5000.00"), and the parse step to strip commas then use CultureInfo.InvariantCulture — making it locale-independent and robust.

Challenge 2 — Shared UI Theme

Problem: Applying consistent colours and fonts across six different form/control files led to repetitive code and inconsistency when tweaking a shade.

Solution: Created ThemeHelper.cs as a central repository of colour constants and factory methods (CreateHeader, CreateStyledGrid, CreatePrimaryButton, etc.). Any theme change is made in one place.

Challenge 3 — Inline Form vs Separate Window

Problem: Deciding whether Add/Edit Student should open a new Form or appear inline in the same control.

Solution: Chose an inline Panel approach — the DataGridView hides and the form panel slides in. This avoids modal dialog management, keeps the code simpler, and gives a smoother UX without extra window chrome.

Challenge 4 — SQLite on First Run

Problem: If the database file doesn't exist, the application would crash with "unable to open database file."

Solution: DatabaseHelper.InitializeDatabase() is called before any form opens. It uses File.Exists + SQLiteConnection.CreateFile and wraps all CREATE TABLE statements in IF NOT EXISTS guards, making the method safe to call on every startup.

8. Trade-offs & Decisions

8.1 SQLite vs SQL Server / MySQL

CriterionSQLite ✅ (Chosen)SQL Server / MySQL
SetupZero — file created at runtimeRequires server install & config
PortabilityOne .sqlite file, copy anywhereMust export/import dump
PerformanceExcellent for < 100k rowsBetter for millions of rows
Concurrent usersSingle-writer (fine for desktop)Multi-user server scenarios
DeploymentNo server neededServer must be running on target

Decision: SQLite is the correct choice for a standalone desktop school admin app. No IT infrastructure is required, making deployment on any school PC trivial.

8.2 ADO.NET vs Entity Framework

CriterionADO.NET ✅ (Chosen)Entity Framework Core
Control over SQLFull — write exactly what you needLINQ translated to SQL (may be inefficient)
Learning curveLow — matches course materialHigher — migrations, DbContext, LINQ
TransparencyEvery SQL statement is visibleSQL hidden behind ORM abstraction
PerformancePredictable, no overheadSlight ORM overhead
BoilerplateMore repetitive connection codeLess; handled by EF

Decision: ADO.NET is chosen because it keeps SQL explicit and testable, aligns with the Visual Programming course focus, and avoids EF migration complexity for a simple schema.

8.3 Windows Forms vs WPF / MAUI

CriterionWindows Forms ✅ (Chosen)WPF / MAUI
Course requirementSpecified in briefNot required
Drag-and-drop designExcellent Visual Studio supportWPF uses XAML (steeper curve)
Maturity25+ years, well-documentedWPF: mature; MAUI: newer
Cross-platformWindows onlyMAUI: multi-platform

Decision: Windows Forms is mandated by the project brief and is the most straightforward choice for a school desktop application that will only run on Windows.

8.4 Static Helper vs Repository Pattern

Decision: A static DatabaseHelper is used instead of a full Repository/Unit-of-Work pattern. For a four-table schema with a single developer, the repository pattern would add substantial boilerplate without measurable benefit. The static helper still cleanly separates data access from UI code, satisfying the modularity requirement.
